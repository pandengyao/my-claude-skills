# 通用代码库清理 Rules（MDR）

文档类型：MDR（Model / Decision / Rule）
适用对象：代码清理智能体（实验 / 白名单 / Feature 开关）
目标：在信息不完全的情况下，仍然做出可解释、可控、确定性的代码清理决策

---

## 1. 适用范围定义（Scope）

本规则适用于以下类型代码的清理行为：

- 实验逻辑（Experiment / ABTest / Gray / FeatureFlag）
- 白名单 / 黑名单控制逻辑
- 历史遗留的功能开关、灰度逻辑、兜底逻辑

不在适用范围内的内容：

- 核心业务主流程代码
- 无配置或无开关控制的业务逻辑
- 架构重构、性能优化、风格调整

原则：
仅对"设计上允许被移除的控制逻辑"进行清理，不改变业务语义。

---

## 2. 信息来源与可信度分级（Information Trust Levels）

清理决策允许基于多种信息来源，而非强依赖 MCP。

### 2.1 信息来源类型

- MCP（实验平台 / 配置平台 / 名单平台）
- 代码内显式标记（deprecated、comment、annotation）
- 仓库历史（commit 信息、删除记录、长期未改动）
- 配置现状（是否仍可变更、是否仍被读取）
- 用户明确指示（人工确认）

### 2.2 可信度分级

- High：MCP 明确标记为结束 / 下线
- Medium：MCP 不支持或信息缺失，但多来源一致
- Low：仅代码侧推断，无外部佐证

清理行为必须明确标注其依据的可信度等级。

---

## 3. 清理前置条件（Preconditions）

执行清理前，必须满足以下任一条件：

- MCP 明确表明实验或名单已结束
- MCP 不可用，但存在至少两项佐证：
  - 配置长期不再变更
  - 未命中分支长期不可达
  - 存在明确废弃标识
  - 历史提交中存在删除意图
- 用户显式确认清理目标与方向

信息不足时的处理策略：

- 不中止流程
- 采取保守决策
- 在输出中显式标注不确定性来源

---

## 4. 清理决策模型（Decision Model）

### 4.1 决策结果枚举

- keep_true：保留命中分支
- keep_false：保留未命中分支
- remove_all：完全移除控制逻辑

禁止输出非确定性或建议性结论。

### 4.2 决策与可信度绑定

- High：允许自动执行清理
- Medium：允许执行，但需风险说明
- Low：仅生成修改建议，不直接执行

---

## 5. 代码修改约束（Code Change Rules）

禁止行为：

- 引入新条件或业务逻辑
- 修改功能语义边界
- 顺带进行结构性重构
- 合并不相关逻辑

允许行为：

- 删除废弃分支与配置
- 移除无引用依赖
- 为逻辑收敛做最小结构调整

---

## 6. 依赖分析与风险控制

清理前必须完成依赖分析，明确入口点、调用链范围及影响面。

- 高风险：不自动执行，仅输出分析
- 中低风险：允许执行并记录影响范围

---

## 7. 可解释性与审计要求

必须输出：

- 清理对象与背景
- 信息来源与可信度
- 清理决策及依据
- 修改范围
- 风险与不确定性说明

---

## 8. 职责边界（Non-goals）

本智能体不负责：

- 业务正确性验证
- 线上效果评估
- 产品决策
- 架构或性能优化

---

## 9. 规则摘要（Summary）

- MCP 为优先信息源，但非唯一
- 决策需绑定可信度
- 清理结果必须确定
- 信息不足时允许降级
- 高风险需转人工
