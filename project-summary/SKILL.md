# 项目总结生成器

根据用户提供的零碎资料，整理生成一份结构化、有说服力的项目总结文档。

## 工作流程

### 第一步：收集资料

向用户确认资料路径。如果用户已经提供了路径，直接读取；否则询问：

- 资料所在目录或文件路径
- 是否有 draft.md 或类似的草稿文件

使用 Glob 和 Read 工具读取指定路径下的所有相关文件（.md、.txt、.doc、.notes 等文本文件）。重点关注：
- draft.md 或任何草稿文件
- 会议记录、设计文档、技术方案
- 数据报告、性能测试结果
- 项目周报、日志

### 第二步：分析和提取

从资料中提取以下关键信息：
- 项目的起因和目标（为什么做这个项目）
- 采用的技术方案和架构选型
- 遇到的难题以及如何解决的
- 可量化的成果数据（性能提升、效率改进、成本节省等）

### 第三步：按大纲撰写

使用以下大纲组织内容：

```
一、项目背景
二、整体技术方案
三、项目难点和挑战
四、项目收益
五、总结
```

各章节要点：

**一、项目背景** — 交代项目的业务场景、痛点问题、项目目标。让读者理解为什么需要这个项目，问题有多迫切。

**二、整体技术方案** — 描述架构设计、技术选型、核心模块。适当使用技术术语体现专业性，但保持逻辑清晰。

**三、项目难点和挑战** — 这是最重要的章节。要把遇到的技术难题、工程挑战写得具体且有深度，然后展示解决方案的巧妙之处。每个难点按「问题描述 → 分析思路 → 解决方案」的结构组织。

**四、项目收益** — 用具体数据说话。性能数据用对比形式呈现（优化前 vs 优化后），业务数据用百分比或绝对值。如果资料中没有具体数字，用合理的描述性语言代替，并用括号标注「待补充具体数据」。

**五、总结** — 简要回顾项目价值，可以提及后续规划或展望。

### 第四步：展示确认

将完整的总结以 Markdown 格式输出给用户审阅。明确告诉用户：
- 可以提出修改意见
- 确认无误后会保存为文件

### 第五步：保存文件

用户确认后，将总结保存为 .md 文件。默认保存在资料同级目录下，文件名为 `项目总结.md`（或用户指定的文件名）。

## 写作风格

- 让读者感受到项目的挑战性和复杂度，但不要夸大其词
- 解决方案的描述要体现思考深度和技术功底
- 数据要具体，避免空泛的「大幅提升」「显著改善」
- 行文流畅自然，避免八股文式的堆砌
- 可以根据资料情况适当调整大纲结构（增删章节），但核心框架保持不变
- 总字数建议在 1500-3000 字之间，视项目复杂度调整
