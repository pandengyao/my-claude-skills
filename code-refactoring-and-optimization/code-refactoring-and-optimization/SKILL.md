# code-refactoring-and-optimization

## 功能概述

本 Skill 专为代码重构与优化场景设计，遵循「先诊断问题、后分层重构、再验证效果」的核心逻辑，优先定位代码的「坏味道」和性能瓶颈，再按「基础重构→模块重构→架构重构」分层落地优化手法，最后通过测试验证功能一致性和性能提升效果，适配遗留系统改造、日常代码优化、技术债务清理等全场景，兼顾「代码质量提升」与「功能安全保障」。

## 适用场景

需要对代码进行重构或性能优化，尤其适用于以下场景：
- 遗留系统维护：代码冗余、嵌套过深、命名混乱，新人接手难以理解
- 性能问题排查：接口响应慢、数据库查询耗时、内存占用高、循环效率低
- 技术债务清理：重复代码多、耦合度高、无测试覆盖，迭代易出bug
- 团队规范落地：代码风格不统一、异常处理混乱，需要标准化重构
- 模块/系统升级：为适配新需求，需先重构现有代码降低扩展成本

|----------------|-------------------------|--------------------------|

## 操作流程

### 1. 确定重构优化核心目标
先通过一句话明确核心目标，聚焦「优化维度+业务范围」，示例：
> 核心目标：优化订单模块calculateOrderAmount函数的可读性（拆分超长逻辑、消除重复代码），同时提升订单金额计算效率（减少循环嵌套、优化优惠券规则查询）

### 2. 诊断代码问题（坏味道+性能瓶颈）
#### 2.1 识别代码坏味道（人工+工具）
列出核心问题，标注问题类型和影响范围：
| 问题类型       | 具体表现                                  | 影响范围               |
|----------------|-------------------------------------------|------------------------|
| 超长函数       | calculateOrderAmount函数共200行，包含金额计算、优惠券校验、运费计算 | 订单模块核心函数       |
| 重复代码       | 优惠券规则校验逻辑在订单创建/修改函数中各出现1次 | 订单创建、订单修改函数 |
| 魔法数值       | 硬编码「满300减50」的300、50等数值         | 金额计算逻辑           |
| 过深嵌套       | if-else嵌套层级达5层（优惠券类型+用户等级+商品分类） | 优惠券校验逻辑         |

#### 2.2 定位性能瓶颈（工具分析）
用性能分析工具（如Arthas、VisualVM、Explain SQL）定位瓶颈，示例：
> 性能瓶颈：
> 1. calculateOrderAmount函数中，优惠券规则查询循环执行10次（N+1查询），单次耗时50ms，占总耗时80%；
> 2. 运费计算逻辑嵌套3层循环，数据量大时耗时超100ms。

### 3. 制定重构/优化方案
按「先重构（提升可读性）、后优化（提升性能）」的顺序制定方案，示例：
| 优化维度       | 具体方案                                  | 预期效果               |
|----------------|-------------------------------------------|------------------------|
| 可读性重构     | 1. 拆分calculateOrderAmount：抽取calculateCouponAmount（优惠券计算）、calculateFreight（运费计算）子函数；<br>2. 魔法数值替换为常量（如FULL_REDUCE_THRESHOLD=300）；<br>3. 简化嵌套：用多态替代优惠券类型的if-else判断 | 函数行数≤50行，嵌套层级≤2层 |
| 性能优化       | 1. 优惠券规则查询：批量查询所有规则，内存中匹配，消除N+1查询；<br>2. 运费计算：减少循环嵌套，用哈希表替代线性查找 | 函数总耗时从200ms降至50ms |

### 4. 小步执行重构（安全优先）
遵循「小步重构、频繁验证、可回滚」原则，分步执行：
