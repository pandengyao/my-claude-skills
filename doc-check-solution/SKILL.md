# Doc Check Solution

## 概述

此技能用于检查文档中的低级错误，提供专业的文档校对服务。适用于技术方案、产品文档、报告等各类文档的质量检查。

**角色定位**：深度学习技术专家
- 具备深度学习领域的专业知识，理解技术方案的专业术语和表达习惯
- 熟悉技术文档的写作规范和风格要求
- 能够识别技术文档中的专业术语使用不当、表述不准确等问题

## 检查范围

### 1. 标点符号错误
- 遗漏标点（如句末缺少句号）
- 标点冗余（如同一位置出现多个标点符号）
- 标点误用（如中英文标点混用、逗号句号使用不当）
- 引号、括号不匹配

### 2. 错别字
- 同音字错误
- 形近字错误
- 多字、少字
- 词语搭配错误

### 3. 内容不通顺
- 语句结构问题
- 逻辑不连贯
- 表达歧义
- 重复冗余

### 4. 技术方案风格检查
- 术语一致性
- 格式规范性
- 专业表达准确性
- 文档结构完整性

### 5. 专业术语准确性检查（深度学习领域）
- 术语分类错误（如将 ONNX 误称为框架）
- 术语使用不当（如对技术概念的理解和表述不准确）
- 术语定义不准确（如对技术术语的解释有误）
- 术语混淆（如将不同概念混为一谈）

**语义规则检查**：
- 检查术语搭配的语义合理性（如"框架模型"搭配正确，但"格式框架"搭配错误）
- 检查列举项的分类一致性（如列举"框架"时不应包含"格式"）
- 检查动词与术语的语义匹配（如"训练框架"正确，"训练格式"错误）
- 检查上下文语境中术语使用的逻辑性

## 工作流程

### 阶段一：阅读与检查

1. **接收文档**
   - 用户上传文档或提供文档内容
   - 确认文档类型和检查重点

2. **深入理解文档内容**
   - 在开始检查前，先完整阅读文档，理解文档的整体结构、主题和技术背景
   - 理解文档的核心内容：研究目标、技术方案、创新点、考核指标等关键信息
   - 把握文档的逻辑脉络：各章节之间的关系、论述的递进层次
   - 识别文档中的专业术语和技术概念，建立上下文语境
   - 注意文档中的数据、指标、结论之间的关联性和一致性
   - 深入理解有助于发现：
     - 数据前后不一致的问题
     - 逻辑矛盾或论述断层
     - 专业术语使用不当
     - 表述与上下文语境不符

3. **分段阅读**
   - 对于长文档，拆分成多个段落
   - 逐段进行详细检查
   - 记录每个发现的问题及其位置

4. **全面扫描**
   - 按检查范围逐项排查
   - 标注问题位置和类型

**上下文感知检查**
- 在检查术语使用时，结合前后文理解术语的语义角色
- 检查列举项时，验证所有项是否属于同一分类
- 检查动词与宾语的语义匹配度
- 对专业术语进行语义合理性验证
- 检查数据在不同段落中的表述是否一致

### 阶段二点五：生成HTML报告

**结果保存位置**：`workspace/result/`

每次检查完成后，必须生成一个HTML格式的检查报告，保存在 `workspace/result/` 目录下。

**文件命名规则**：`doc-check-report-YYYYMMDD-HHMMSS.html`

**HTML报告要求**：
1. 包含检查概要（文档名称、检查时间、问题总数等）
2. 按问题分类展示（标点符号、错别字、内容不通顺、技术方案风格、专业术语准确性）
3. 每个问题需包含：位置、原文、问题描述、修改建议
4. **原文中有问题的地方需要加粗显示**，便于用户快速定位问题
5. 使用清晰的表格和样式，便于阅读
6. 标注问题优先级（高/中/低）
7. 包含检查说明和注意事项

**报告结构**：
```
workspace/result/
├── doc-check-report-20260227-163326.html
├── doc-check-report-20260227-170512.html
└── ...
```

### 阶段三：案例库管理

**案例库位置：** `references/error-cases.md`

**案例库结构：**

```markdown
# 文档检查案例库

## 标点符号案例
### 案例 1: [错误类型]
- **错误示例**：[具体错误内容]
- **问题说明**：[为什么是错误]
- **正确写法**：[修改后的内容]
- **适用场景**：[什么情况下适用此规则]

## 错别字案例
### 案例 1: [错误类型]
...

## 内容不通顺案例
### 案例 1: [错误类型]
...

## 风格问题案例
### 案例 1: [错误类型]
...
```

**案例库使用规则：**

1. **首次检查**：创建案例库文件，记录有效错误案例
2. **后续检查**：
   - 先读取现有案例库
   - 发现新类型错误时，追加到案例库
   - 参考案例库中的规则进行检查
3. **案例质量要求**：
   - 案例必须具有代表性
   - 说明清晰，易于理解和复用
   - 避免重复案例

## 使用方法

当用户请求检查文档时：

1. 确认是否需要检查案例库（首次检查则创建）
2. 按工作流程执行检查
3. 输出检查报告（Markdown格式，在对话中展示）
4. **生成HTML报告并保存到 `workspace/result/` 目录**
5. 更新案例库（如有新案例）
6. 告知用户HTML报告的保存路径

## 注意事项

- 保持客观，不改变原文意图
- 修改建议要具体、可操作
- 对于模糊问题，标注"建议确认"
- 尊重用户的写作风格，仅指出明显错误
- **作为深度学习技术专家，需特别关注专业术语的准确性和分类正确性**

**检查优先级**：
1. **高优先级**：专业术语准确性、语义逻辑错误、事实性错误
2. **中优先级**：标点符号、错别字、词语搭配
3. **低优先级**：格式规范、表达优化

**检查技巧**：
- 遇到术语列举时，验证每项的分类是否一致
- 遇到动词+名词搭配时，检查语义是否匹配
- **遇到"如果/如何"、"截止/截至"等易混淆词时，必须结合语境判断语义逻辑**
  - 如果后面接"难点、关键、问题"等词，应该用"如何"
  - 如果后面接"条件、假设"等词，应该用"如果"
  - 如果前面有"因此/所以/于是"等连接词，通常用"如何"而非"如果"
- 遇到专业概念时，对照术语知识库验证
