# 论文精读 (Paper Reader)

对学术论文（以AI/ML为主）进行深度阅读，生成全面的结构化阅读笔记，保存为Markdown文件。支持单篇和批量输入。

## 文件结构

本 skill 由 3 个文件组成：
- **SKILL.md**（本文件）：主流程、精读各阶段、输出规范
- **image-extraction.md**：图表提取的完整代码与验证流程（~1000行）
- **note-template.md**：笔记 Markdown 模板结构（~200行）

## 工作流程

```
输入解析 → 多工具提取与交叉验证 → 第一遍速览 → 第二遍精读 → 第三遍批判 → 正确性验证 → 输出生成（含回读验证）
```

**多工具提取**：根据PDF类型智能选择工具组合（详见 `image-extraction.md`），通过优先级规则融合多源信息，防止信息丢失，生成完整的信息全集后再进入精读流程。

批量输入时，对每篇论文独立执行完整流程，生成一一对应的阅读笔记文件。


## 第一阶段：输入解析

### 单篇输入

#### arXiv URL

当用户提供arXiv链接（如 `https://arxiv.org/abs/2401.xxxxx` 或 `https://arxiv.org/pdf/2401.xxxxx`）：

1. 从URL提取论文ID
2. 用WebFetch获取arXiv摘要页，提取标题、作者、摘要等基本信息
3. 构造PDF链接：`https://arxiv.org/pdf/{paper_id}.pdf`
4. 用 `curl -L -o` 下载PDF到 `papers/pdf/` 目录（已存在同名文件时直接覆盖）：
```bash
curl -L -o papers/pdf/{paper_id}.pdf https://arxiv.org/pdf/{paper_id}.pdf
```
   （同样适用于GitHub等其他来源的PDF链接）
5. 用Read工具读取PDF内容

#### 本地PDF

当用户提供文件路径：

1. 确认文件存在且为PDF格式
2. 用Read工具直接读取（大型PDF超过10页时分批读取，每次≤20页）

### 批量输入

当用户提供多个链接或文件时（以下形式均支持）：
- 对话中直接贴多个 arXiv URL（每行一个或空格/逗号分隔）
- 对话中列出多个本地 PDF 路径
- 提供一个包含链接列表的文本文件（.txt，每行一个URL或路径）
- 混合提供 arXiv URL 和本地 PDF

处理方式：
1. 先解析出所有论文的链接/路径列表
2. 向用户确认识别到的论文数量和列表
3. **逐篇**执行完整的精读流程（速览→精读→批判→验证→输出）
4. 每篇论文生成独立的阅读笔记文件，文件名与论文一一对应
5. 全部完成后，向用户汇报所有生成的笔记文件路径

### 读取失败处理

```
PDF读取遇到问题。可以尝试：
1. 确认文件路径正确且文件未损坏
2. 对于arXiv论文，可提供abs页面链接，我会同时参考网页信息
3. 如有论文的文本版本（如LaTeX源码），也可以直接提供
```


## 第二阶段：多工具提取与交叉验证

**完整流程详见 `image-extraction.md`。** 以下是关键要点摘要：

### 核心理念
根据PDF类型智能选择工具组合，用优先级规则融合多源信息。所有Python脚本使用 `~/py312/bin/python` 执行。

### 工具箱（按优先级）
| 工具 | 用途 | 必选? |
|------|------|-------|
| Read工具 | Claude原生PDF读取 | 必选 |
| PyMuPDF | 高精度文字+结构提取 | 必选 |
| pdf2image + VLM | 关键页面视觉理解 | 必选 |
| pdfplumber/camelot | 线框表格提取 | 仅Word/排版PDF |
| Surya | 布局检测（Figure精确定位） | 图片提取优先方案 |
| marker-pdf | ML全能转换 | 可选补充 |

### PDF 类型判断
- **LaTeX学术论文**（arXiv等）→ Read + PyMuPDF + pdf2image（跳过 pdfplumber/camelot）
- **Word/排版工具PDF** → 全工具启用
- **扫描件/图片PDF** → Read + pdf2image + marker-pdf

### 信息融合优先级
1. 图表和公式 → 视觉理解优先
2. 正文文本 → PyMuPDF 为主
3. 表格数据 → pdfplumber/camelot（若有）> 视觉理解
4. 遇到分歧 → 以视觉理解为最终裁判

### Context 预算控制
- PyMuPDF 全文保存到 `/tmp/paper_pymupdf_text.txt`，按需分段读取
- pdf2image 仅转换关键页面（通常不超过 10 页）
- 进入精读阶段前，仅保留融合后的信息全集摘要

### 注意事项
- 工具执行失败时跳过继续，不阻塞整体流程
- 临时文件统一放 `/tmp/`，笔记生成完成后清理
- 论文图片保存到 `papers/images/{paper_id}/`
- **图片压缩（必做）**：提取完成后，将所有图片 resize 到 max_width=800px，转为 JPEG quality=85 后再 base64 嵌入笔记。目标：单张图 < 100KB，笔记总文件 < 1MB

### 图片提取强制执行规则（MANDATORY — 不可跳过）

**核心原则：禁止盲猜坐标。必须先分析再裁剪，先验证再使用。**

图片提取必须严格遵循以下三步闭环流程，任何一步都不可省略：

#### Step A：坐标分析（禁止凭感觉估计）
提取任何图片前，**必须**先用 PyMuPDF 的 `get_text("dict")` 获取页面所有 text block 的精确 bbox，找到：
1. Caption text block 的精确 y 范围（包含所有续行）
2. 嵌入图片（`get_images()` + `get_image_rects()`）或矢量图（`get_drawings()`）的精确位置
3. Caption 下方第一个正文 text block 的 y0（作为裁剪下界）

**禁止行为**：使用 `fitz.Rect(72, "估计值", 560, "估计值")` 这种凭页面比例猜测的方式。

#### Step B：精确裁剪
基于 Step A 获得的精确坐标进行裁剪：
- y0 = 图形元素（images/drawings/legend text）的最小 y - 5pt
- y1 = caption 最后一行 bbox 的 y1 + 3pt（但不超过下一个正文 block 的 y0 - 3pt）
- x 范围确定规则（按优先级）：
  1. 计算图形区域内所有元素（drawings + images + 短文本标签）的实际 x 范围
  2. 取 max(caption x 范围, 图形元素 x 范围) 作为最终 x 范围，再外扩 5pt
  3. **注意**：图内标签（如方法名 "EAGLE-3"、公式标注 "$l_{\text{fea}}$ removal"）经常超出正文列宽。不能简单使用"所在列宽度"作为 x 范围，必须检查图内所有 text span 的实际 x1 取最大值

#### Step C：视觉验证（模拟人类 Reviewer 逐边检查，必做）
提取完所有图片后，**必须**用 Read 工具逐一查看每张图片，**模拟人类 reviewer 的逐边审查过程**：

**四边检查法（按顺序逐边扫描）**：
1. **上边缘**：图片最顶部是否属于该图/表？有没有混入论文标题、作者信息、上方段落的尾行？
2. **下边缘**：图片最底部是否属于该图/表？有没有混入下一节标题、正文段落的首行？Caption 是否完整（最后一行没有被截断）？
3. **左边缘**：有没有截入相邻列的内容（双栏论文）？图内最左侧元素是否完整？
4. **右边缘**：图内最右侧的标签/注释是否完整？有没有文字被截断（如只显示了"remov"而非"removal"）？有没有截入右栏内容？

**内容完整性检查**：
5. 图中所有**文字标签**是否可读且完整（坐标轴标签、图例、方法名、公式标注）？
6. Caption 全文是否可读且无截断（特别注意多行 caption 的最后一行）？

**判断标准**：对每张图回答"四边有没有多余/缺失？"——如果任何一边有问题，回到 Step A 重新分析。

**关键心理模型**：想象你是用户第一次看到这张截图。你会不会觉得"这上面的标题怎么混进来了"或"右边这个词怎么只有一半"？如果会，就说明截取不精确。

**若发现问题**：回到 Step A 重新分析该图的坐标，修正后重新裁剪。最多循环 2 次，第 3 次使用整页截图兜底。

**为什么必须这样做**：
- 第一遍盲猜坐标的错误率 > 50%（常见问题：底部混入正文、图形被截断、caption 续行遗漏）
- 视觉验证是唯一可靠的质量保证手段
- 这个 closed-loop 流程模拟了"人类发现问题 → 指出修正"的反馈过程，将其内化为自动执行的步骤


## 第三阶段：第一遍速览（Survey）

快速浏览，建立整体认知框架。重点关注：

- **标题与摘要**：提炼核心主张
- **引言**：明确问题定义和研究动机
- **章节标题与图表**：把握论文结构和关键结果
- **结论**：确认主要贡献声明

完成后填写：论文基本信息、一句话总结、研究动机、核心贡献列表。


## 第四阶段：第二遍精读（Comprehension）

逐节深入阅读，重点理解：

**详细度要求**：核心章节（方法部分、实验部分）的笔记内容应达到近乎翻译的详细程度，逐段忠实呈现原文的技术细节和论述逻辑，不做跳跃式概括。

### 方法部分
- 问题形式化定义（数学符号、变量含义）
- 模型/算法架构设计及其动机
- **所有公式的完整推导**（逐步展开，不跳步）
- 与已有方法的核心区别

### 实验部分
- 数据集选择及其合理性
- 基线方法（Baselines）的选取是否公平全面
- 评价指标的选择
- 主实验结果及消融实验（Ablation Study）的关键发现
- 实验设置中的重要超参数

### 关键图表
- 对每个重要图表进行解读
- 指出图表揭示的关键趋势或结论


## 第五阶段：第三遍批判（Critique）

以审稿人（Reviewer）视角进行批判性分析：

### 创新点评估
- 与相关工作的详细对比（方法层面，非简单罗列）
- 创新性类型判断：全新范式 / 有效组合 / 增量改进
- 创新点是否足够支撑论文的核心主张

### 优势分析
- 方法设计的优雅之处
- 实验的充分性和说服力
- 理论或实践意义

### 局限性分析
- 假设条件是否过强
- 实验覆盖是否有盲区
- 可扩展性（Scalability）问题
- 论文中未讨论但值得注意的局限

### 实践导向提炼
- **算法伪代码**：将核心算法整理为清晰的伪代码（若论文已提供则优化注释）
- **复现要点**：关键实现细节（特殊初始化、训练技巧、数据预处理等）
- **代码资源**：论文提供的开源代码链接、第三方复现仓库、相关工具库


## 第六阶段：正确性验证（Verification）

对论文中的公式、数据和结论进行验证，这是区别于普通阅读笔记的关键环节：

### 公式验证
- 检查推导过程是否有跳步或错误
- 验证维度/量纲是否一致
- 检查边界条件和特殊情况是否合理
- 用具体数值代入验证公式的正确性

### 数据验证
- 核对表格中的数据是否与文中描述一致
- 检查百分比计算、数值大小是否合理
- 关注不同表格/图之间数据是否自洽

### 结论验证
- 实验结果是否真正支撑核心结论
- 是否存在过度解读或以偏概全
- 统计显著性是否充分

每项验证结果标注为：✅ 已验证 / ⚠️ 存疑 / ❌ 发现问题


## 第七阶段：输出生成

### 文件命名与保存

确定输出路径（按优先级）：
1. `/Users/frank/work/papers/`（默认目录）,有此目录就放进去，没有也不要创建
2. 用户指定的目录
3. PDF所在同目录
4. 当前工作目录

文件名格式：`{第一作者姓氏}_{年份}_{简短标题关键词}_阅读笔记.md`
- 示例：`Vaswani_2017_Attention_阅读笔记.md`
- **重复精读**：若目标路径已存在同名笔记文件，直接覆盖（重新精读以最新结果为准）。同时覆盖 `papers/images/{paper_id}/` 下的旧图片。

### 笔记模板

**完整模板结构详见 `note-template.md`。** 用中文撰写全部内容，但技术术语尽可能保留英文原文（参见下方"术语翻译原则"）。

笔记包含以下章节：
1. TL;DR（一句话总结 + 核心创新 + 关键数字）
2. 摘要（近乎逐句翻译）
3. 论文基本信息表
4. 研究动机与问题定义
5. 方法详解（含完整公式清单与数值示例、算法伪代码）
6. 实验分析（设置、主要结果、消融、图表解读）
7. 批判性分析（创新点评估、优势、局限、相关工作对比）
8. 正确性验证报告
9. 复现指南
10. 延伸思考
11. 核心参考文献（6-12篇，按主题分组，每篇必须附 arXiv/DOI 链接）


## 重要注意事项

### 内容质量
- **准确性优先**：不确定的内容标注"（待确认）"，绝不编造论文未提及的内容
- **公式完整性**：LaTeX公式确保可正确渲染，每个公式都要配数值示例
- **术语一致性**：全篇保持专业术语一致
- **术语翻译原则（重要！）**：默认保留英文原文，仅对少数有公认中文术语的概念使用中文。具体规则：
  - **必须保留英文的术语**（不完全列表）：token, logits, batch size, fan-out, lookahead, draft, target, cache, hit/miss, speculate/speculation, verify/verification, fallback, sampling, throughput, latency, backbone, head, attention, embedding, inference, decoding, prefill, forward pass, residual distribution, acceptance rate, bonus token 等
  - **可以翻译的术语**（有公认中文的）：梯度(gradient)、损失函数(loss)、权重(weight)、训练(training)、过拟合(overfitting)、卷积(convolution)、池化(pooling) 等
  - **判断标准**：如果一个英文术语在中文ML社区口语交流中通常直接说英文（如"这个token的logits"而非"这个词元的对数概率"），则保留英文
  - **中英混排格式**：自然嵌入中文句子中，如"draft model在verification进行时预计算多个speculation branch"
- **批判性但公正**：指出局限时要具体、建设性，避免空洞批评
- **摘要与核心章节近乎翻译**：对于论文摘要和核心章节（方法详解、实验分析），笔记内容应达到近乎逐句翻译的详细程度，忠实保留原文的所有技术细节、数据和论述逻辑
- **图片编号顺序优先（强制）**：笔记中 Figure/Table 必须严格按自然编号递增顺序出现（Figure 1 在 Figure 2 之前，Figure 6 在 Figure 7 之前）。即使原文在后面章节才首次引用某个前序编号的图（如 Figure 1 出现在 Discussion 中），也必须将其提前到笔记更早的位置（如研究动机/引言部分），确保读者看到的图片序号始终递增。**写笔记前必须先规划所有 Figure/Table 的插入位置，确认编号递增后再开始撰写。**
- **图片提取精度**：首次自动提取后必须逐一视觉验证，发现截断/多余内容时通过坐标诊断精确修正
- **内容纯粹性**：最终笔记只包含论文内容分析，不暴露工具提取过程等元信息

### 图表提取与排版三原则（强制要求）

**原则一：精确截取，不多不少**
- 每张图/表的截取区域必须**仅包含该图/表本身及其完整 caption（含所有续行）**，不得包含正文文字、相邻图/表内容或页眉页脚
- **坐标确定方法（必须遵循）**：
  1. 用 PyMuPDF `get_text("dict")` 获取页面所有 text block 的精确 bbox
  2. 找到目标 Figure/Table 的 caption text block，记录其 bbox 的 y 范围
  3. **x 范围必须使用全文本列宽**（即页面 text area 的左右边界，通常为 LEFT_MARGIN ~ RIGHT_MARGIN）。绝不能使用 figure/table 本体的居中宽度作为 x 范围——caption 文字通常比图/表本体更宽，使用图/表本体宽度会导致 caption 右侧被截断
  4. **caption 底部确定（关键！）**：Caption 经常是多行的，PyMuPDF 可能将其报告为单个 text block 或多个 block。必须搜索 caption 起始行下方所有属于该 caption 的续行 text block，取最终底部 y + 3~5pt 作为裁剪 y1
  5. 裁剪 y0 的确定：
     - **对于 Figure**：优先使用 `get_images()` 获取嵌入图片的精确 y0；若纯矢量图，用 `get_drawings()` 的最小 y0；**还必须搜索 figure 区域内的短文本标签**（如坐标轴标签 "A > 0"、"L^CLIP" 等，通常 len < 20），将这些标签的 y0 也纳入裁剪上界计算
     - **对于 Table**：从 caption block 的 y0 向上搜索表格内容
  6. **表格特别注意**：表格末行数据的 bbox 底部 + 10pt 才是安全下边界（需包含底部横线/rule）
  7. 找到 caption 下方最近的**正文** text block，确保裁剪 y1 不超过该 block 的起始 y - 3pt
- **常见陷阱（按发生频率排序）**：
  - **Caption 右侧截断**（最常见！）：x 范围使用了图/表本体宽度而非全列宽。**修复**：始终用 LEFT_MARGIN ~ RIGHT_MARGIN
  - **Caption 底部截断**：多行 caption 仅取了第一行的 bbox。**修复**：搜索 caption 所有续行
  - **Figure 顶部标签截断**：仅用 drawings/images 的 bbox，忽略了图内文字标签。**修复**：搜索 figure 区域内 len < 20 的文本 block
  - **混入正文段落**：caption 下方的正文被误判为 caption 续行。**修复**：正文段落通常以大写字母开头且长度 > 100 字符
  - 表格底部 rule line 在最后一行文字 bbox 下方 5-10pt
  - 两栏论文中 Figure caption 若跨行，bbox 可能比预期更高
  - **章节标题误截入**：正确做法是用 `get_images()`/`get_drawings()` 定位图形元素实际顶部
- **验证-修正循环**：提取 → 视觉验证 → 修正坐标 → 重新提取 → 再验证，直到每张图都精确

**原则二：全量提取，从 Table 1 开始**
- 论文中**所有**编号的 Figure 和 Table 都必须提取为图片并嵌入笔记
- 不可跳过任何编号的图/表

**原则三：图文对应，顺序连贯（自然编号优先）**
- **核心约束：自然编号顺序优先**。笔记中 Figure/Table 必须按编号递增顺序出现
- 在满足编号顺序的前提下，每张图/表尽量紧跟首次引用它的正文段落之后
- **冲突处理**：将前序编号的图提前到更早章节（如将概览图放到引言），确保编号递增
- **常见错误示例**：论文在 Discussion 章节才首次引用 Figure 1（如总览时间线图），而 Figure 2 在方法章节引用。如果按引用顺序放置，笔记中会出现 Fig2 → Fig1 的倒序。正确做法是将 Figure 1 放到研究动机/引言部分（语义上也合适），保证 Fig1 → Fig2 递增
- 图/表下方必须有中文翻译的完整 caption（加粗格式）
- 禁止将所有图/表集中堆放在某一个小节中
- **操作流程**：先按编号列出所有 Figure/Table 的语义和原文引用位置，规划插入位置，确保编号递增后再开始正文撰写

### 适配不同论文类型

根据论文类型调整精读重点和**裁剪笔记模板**：

- **理论型论文**：强化公式清单和正确性验证，可简化实验分析和复现指南
- **系统型论文**：强化核心技术细节和复现指南，可简化公式清单
- **实证型论文**：强化实验分析和数据验证，可简化算法伪代码
- **综述型论文**：省略公式清单、伪代码、正确性验证、复现指南，增加"领域全景图"和"研究趋势"

### 与用户交互
- 生成笔记后询问用户是否需要对某个部分深入展开
- 如论文涉及不熟悉的领域知识，主动说明并建议用户补充确认
- 对于特别长的论文（>30页），先告知预计阅读范围，确认后再进行

### 输出操作
1. 用Write工具将完整Markdown写入目标文件
2. **格式校验清单（MANDATORY — 写入后立即逐项检查）**：
   回读笔记前 20 行和最后 30 行，逐项确认以下格式要素完整：
   - [ ] 第 1 行：英文标题（`# {英文标题}`）
   - [ ] 第 2 行：中文标题翻译（`# {中文标题}`）
   - [ ] TL;DR 包含：一句话总结 + 核心创新 + 关键数字

   - [ ] 全文无 `---` 分隔线（章节间用空行分隔，不用水平线）
   - [ ] 所有图片使用 base64 内联嵌入（`<img src="data:image/png;base64,..."`），禁止使用相对路径引用（`![](path)`）
   - [ ] Figure/Table 编号严格递增（grep `<img.*alt="Figure` 确认顺序）
   - [ ] 每张图/表下方有加粗中文 caption

   - [ ] 若任一项缺失，立即修正后重新写入

   - 2-3个核心公式的LaTeX是否能正确渲染且与原文一致
   - 作者、机构、发表venue等基本信息是否准确
   - 若发现不一致，立即修正后重新写入
4. 告知用户文件保存路径
