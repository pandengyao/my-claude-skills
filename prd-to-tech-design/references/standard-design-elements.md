# 标准设计元素参考

本文档列出在获得充分上下文后应该自主补充到技术设计文档中的标准设计元素，无需反复询问用户。

## 何时使用本文档

在完成基本技术设计后，使用本文档作为完整性检查清单，系统性地补充标准设计元素。

## 完整性检查清单

- [ ] **错误处理**：是否定义了完整的错误码？是否设计了统一的错误响应格式？
- [ ] **日志记录**：是否覆盖了所有关键操作？是否定义了日志级别和脱敏规则？
- [ ] **监控指标**：是否设计了接口监控、业务监控、资源监控？
- [ ] **数据校验**：是否定义了所有必填字段、格式校验、长度限制？
- [ ] **数据库设计**：是否包含标准字段（id、created_at、updated_at、deleted_at）？是否设计了索引？
- [ ] **接口设计**：是否包含标准参数（分页、排序、过滤）？是否考虑了幂等性？
- [ ] **安全防护**：是否考虑了 SQL 注入、XSS、SSRF、鉴权、加密？
- [ ] **性能优化**：是否设计了索引、缓存、分页、异步处理？
- [ ] **测试方案**：是否列出了单元测试、接口测试、边界测试场景？
- [ ] **运维部署**：是否提供了数据库迁移脚本、配置项、回滚方案？

## 1. 错误处理机制

### 1.1 统一错误码定义

在"关键点设计"章节中补充完整的错误码定义表：

| 错误码 | 说明 | HTTP 状态码 |
|--------|------|-------------|
| 0 | 成功 | 200 |
| 40001 | 参数校验失败 | 400 |
| 40101 | 未授权 | 401 |
| 40301 | 无权限 | 403 |
| 40401 | 资源不存在 | 404 |
| 50001 | 服务器内部错误 | 500 |

### 1.2 错误响应格式

```json
{
  "code": 40001,
  "message": "参数校验失败：用户名不能为空",
  "data": null
}
```

### 1.3 异常处理策略

- 统一异常捕获和降级处理
- 用户友好的错误提示
- 关键错误记录日志并告警

## 2. 日志记录设计

### 2.1 关键操作日志记录点

- 创建操作：记录实体 ID、创建人、创建时间、关键字段
- 更新操作：记录实体 ID、修改人、修改时间、修改前后值
- 删除操作：记录实体 ID、删除人、删除时间、删除原因
- 查询操作：记录查询条件、结果数量、耗时
- 外部调用：记录请求参数、响应结果、耗时、状态码

### 2.2 日志级别定义

- **INFO**：正常操作（创建、更新、查询成功）
- **WARN**：异常情况（校验失败、重试、降级）
- **ERROR**：错误情况（系统异常、外部服务失败、数据不一致）

### 2.3 敏感信息脱敏规则

- 密码：记录为 "******"
- Token：记录前 8 位 + "..."
- 手机号：记录前 3 位 + "****" + 后 4 位
- 身份证号：记录前 6 位 + "********" + 后 4 位
- 银行卡号：记录前 4 位 + "****" + 后 4 位

### 2.4 日志格式

```
[时间戳] [日志级别] [模块名] [操作类型] [用户ID] [详细信息]
```

## 3. 监控和告警指标

### 3.1 接口监控

- 接口响应时间（P50/P95/P99）
- 接口错误率（按错误码分类）
- 接口 QPS（每秒请求数）
- 接口超时率

### 3.2 业务监控

根据具体业务设计关键指标，例如：
- 任务成功率
- 用户操作成功率
- 数据处理成功率
- 外部服务调用成功率

### 3.3 资源监控

- 数据库连接数
- Redis 缓存命中率
- CPU 使用率
- 内存使用率
- 磁盘使用率
- 队列积压数

### 3.4 告警规则

- 接口错误率 > 5% 持续 5 分钟 → 告警
- 接口响应时间 P95 > 1s 持续 5 分钟 → 告警
- 数据库连接数 > 80% 持续 3 分钟 → 告警
- 队列积压 > 1000 持续 10 分钟 → 告警

## 4. 数据校验规则

### 4.1 必填字段校验

列出所有必填字段，在接口层进行校验。

### 4.2 数据格式校验

- 邮箱：正则表达式校验
- 手机号：正则表达式校验（支持国际格式）
- URL：格式校验（http/https）
- 日期时间：格式校验（ISO 8601）
- JSON：格式校验

### 4.3 数据长度限制

- 字符串字段：定义最小/最大长度
- 数组字段：定义最小/最大元素数
- 文件上传：定义最大文件大小

### 4.4 业务规则校验

根据具体业务定义校验规则，例如：
- 时间范围：开始时间 < 结束时间
- 数量限制：数量 > 0
- 唯一性：用户名/邮箱不重复
- 依赖关系：删除前检查是否有关联数据

## 5. 数据库设计标准

### 5.1 标准字段

所有表都应包含以下标准字段：

```sql
id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
deleted_at TIMESTAMP NULL COMMENT '软删除时间',
created_by BIGINT COMMENT '创建人ID',
updated_by BIGINT COMMENT '更新人ID'
```

### 5.2 索引设计原则

- 为高频查询字段创建索引
- 为外键字段创建索引
- 为联合查询创建联合索引（注意字段顺序）
- 避免过多索引影响写入性能

### 5.3 字段设计规范

- 使用有意义的字段名（英文小写 + 下划线）
- 为每个字段添加 COMMENT 注释
- 选择合适的数据类型（避免过度设计）
- 定义字段约束（NOT NULL、DEFAULT、UNIQUE）

## 6. 接口设计标准

### 6.1 标准分页参数

```json
{
  "page": 1,
  "page_size": 20,
  "total": 100
}
```

### 6.2 标准排序参数

```json
{
  "sort_by": "created_at",
  "order": "desc"
}
```

### 6.3 标准过滤参数

```json
{
  "filter": {
    "status": "active",
    "created_at_start": "2024-01-01",
    "created_at_end": "2024-12-31"
  }
}
```

### 6.4 请求幂等性设计

对于创建/更新操作：
- 使用幂等键（idempotency_key）
- 相同幂等键的重复请求返回相同结果
- 幂等键有效期 24 小时

### 6.5 接口版本控制

使用 URL 路径版本控制：
- `/api/v1/users`
- `/api/v2/users`

## 7. 安全防护措施

### 7.1 输入验证

- 所有用户输入进行白名单校验
- 拒绝包含特殊字符的输入（除非业务需要）
- 限制输入长度

### 7.2 SQL 注入防护

- 使用参数化查询（PreparedStatement）
- 禁止拼接 SQL 语句
- 使用 ORM 框架（如 SQLAlchemy、MyBatis）

### 7.3 XSS 防护

- 输出到前端的数据进行 HTML 转义
- 使用 Content-Security-Policy 头
- 避免使用 innerHTML，使用 textContent

### 7.4 SSRF 防护

参考项目 CLAUDE.md 中的安全规范：
- 禁止直接使用前端传入的 URL
- 后端校验 URL 白名单
- 禁止访问内网地址

### 7.5 接口鉴权

- 所有接口需要验证用户登录态
- 使用 JWT 或 Session 进行身份验证
- 实现基于角色的权限控制（RBAC）

### 7.6 敏感数据加密

- 密码使用 bcrypt/scrypt 加密存储
- 敏感配置（API Key、密钥）使用 AES 加密存储
- 传输层使用 HTTPS

## 8. 性能优化方案

### 8.1 数据库查询优化

- 为高频查询字段创建索引
- 避免 SELECT *，只查询需要的字段
- 避免 N+1 查询问题
- 使用批量操作代替循环单条操作
- 分页查询限制每页最多 100 条

### 8.2 缓存策略

- 使用 Redis 缓存热点数据
- 定义合理的缓存过期时间（TTL）
- 缓存 key 命名规范：`模块:实体:ID`
- 更新/删除时主动清除缓存
- 使用缓存预热避免冷启动

### 8.3 分页加载

- 列表接口必须支持分页
- 默认每页 20 条，最大 100 条
- 使用游标分页（cursor-based）处理大数据集

### 8.4 异步处理

- 长时间操作（> 3 秒）后台任务化
- 使用消息队列（如 RabbitMQ、Kafka）
- 提供任务状态查询接口
- 任务失败支持重试机制

### 8.5 CDN 加速

- 静态资源（图片、CSS、JS）使用 CDN
- 配置合理的缓存策略
- 支持 HTTPS

## 9. 测试方案要点

### 9.1 单元测试

- 覆盖核心业务逻辑
- 覆盖边界条件和异常情况
- 使用 Mock 隔离外部依赖
- 目标覆盖率 > 80%

### 9.2 接口测试

- 测试正常流程
- 测试参数校验（必填、格式、长度）
- 测试权限控制
- 测试并发场景

### 9.3 边界条件测试

- 空值、null、空字符串
- 最大值、最小值
- 特殊字符
- 超长输入

### 9.4 性能测试

- 压力测试（模拟高并发）
- 负载测试（持续负载）
- 响应时间测试
- 数据库连接池测试

## 10. 运维和部署考量

### 10.1 数据库迁移脚本

提供完整的 DDL 脚本：
- 建表语句
- 索引创建语句
- 初始化数据语句
- 版本号和变更说明

### 10.2 配置项管理

- 使用配置中心（如 Apollo、Nacos）
- 区分环境配置（dev/test/prod）
- 敏感配置加密存储
- 配置变更支持热更新

### 10.3 灰度发布策略

- 按用户 ID 百分比灰度
- 按地区灰度
- 支持快速回滚
- 监控灰度期间的关键指标

### 10.4 回滚方案

- 代码回滚：回退到上一个稳定版本
- 数据回滚：提供数据恢复脚本
- 配置回滚：恢复旧配置
- 回滚验证：验证回滚后系统正常

## 何时不需要补充

以下情况可以省略某些标准元素：
- PRD 明确说明不需要某个功能（如"不需要软删除"）
- 技术方案本身不涉及某个领域（如纯前端项目不需要数据库设计）
- 用户明确表示"简化设计"或"快速原型"
