# 示例与常见错误

本文档提供精简的示例,展示正确做法和常见错误。

## 1. 证据扫描示例

### ✅ 正确做法:适度扫描,及时澄清

```
用户:为这个 PRD 生成技术设计
[PRD 提到"唤起账号登录器"]

AI:我先扫描一下现有实现...
[扫描 src/auth/、components/Login/]

AI:证据扫描结果:
| 能力点 | 证据路径 | 当前实现 | 缺口/冲突 |
|--------|----------|----------|-----------|
| 基础登录 | src/auth/login.py | 已实现 | - |
| 登录器唤起 | - | 未找到 | 需确认 |

PRD 提到"唤起账号登录器",但在代码中未找到相关实现。请问:
1. 登录器是前端组件还是独立客户端?
2. 是否有相关的技术文档或代码文件可以参考?
```

### ❌ 错误做法:盲目探索代码库

```
用户:为这个 PRD 生成技术设计
AI:让我先探索代码库,了解现有的账号管理实现...
[无差别扫描整个代码库,浪费时间]
```

**问题**:代码可能过时、浪费时间、可能误解实现

## 2. 技术不确定性处理示例

### ✅ 正确做法:主动请求明确参考信息

```

2. 如何触发登录流程?

请提供:文字说明、文档路径、或关键代码文件路径
```

### ❌ 错误做法:盲目假设技术可行性

```
用户:PRD 需要支持定时发布
AI:我将使用 APScheduler 实现定时发布功能...
```

**问题**:不确定 APScheduler 是否支持动态任务调度、是否已使用、是否适合

**场景:用户已提供 PRD 和关键技术参考信息**

```

[用户提供了 PRD 和现有系统的技术文档]

[AI 在设计文档中自主补充以下内容,无需询问用户:]

四、设计思路与实现方案

### 2.1 数据库设计

**账号配置表(account_config)**
```sql
CREATE TABLE account_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL COMMENT '用户ID',
    platform VARCHAR(50) NOT NULL COMMENT '平台类型',
    account_name VARCHAR(100) NOT NULL COMMENT '账号名称',
    proxy_config JSON COMMENT '代理配置',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    deleted_at TIMESTAMP NULL COMMENT '软删除时间',
    INDEX idx_user_platform (user_id, platform)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.2 接口设计

**创建账号配置**
- 路径:POST /api/v1/accounts
- 请求参数:
  - user_id (必填): 用户ID
  - platform (必填): 平台类型
  - account_name (必填): 账号名称,长度 1-100
- 响应:
  - 成功:200 { "code": 0, "message": "success", "data": { "id": 123 } }
  - 失败:400 { "code": 40001, "message": "参数校验失败" }

### 2.3 错误码定义

| 错误码 | 说明 | HTTP 状态码 |
|--------|------|-------------|
| 0 | 成功 | 200 |
| 40001 | 参数校验失败 | 400 |
| 40101 | 未授权 | 401 |
| 50001 | 服务器内部错误 | 500 |

### 2.4 日志记录策略

**关键操作日志:**
- 账号创建:记录 user_id、platform、account_name、操作时间
- 账号更新:记录 account_id、修改字段、操作时间
- 账号删除:记录 account_id、删除原因、操作时间

**日志级别:**
- INFO: 正常操作
- WARN: 异常情况
- ERROR: 错误情况

### 2.5 监控指标设计

**接口监控:**
- 接口响应时间(P50/P95/P99)
- 接口错误率
- 接口 QPS

**业务监控:**
- 账号创建成功率
- 登录器唤起成功率

### 2.6 安全防护措施

- **输入验证**:所有用户输入进行白名单校验
- **SQL 注入防护**:使用参数化查询
- **XSS 防护**:输出数据进行 HTML 转义
- **接口鉴权**:所有接口验证用户登录态
- **敏感数据加密**:代理密码使用 AES 加密存储

### 2.7 性能优化方案

- **数据库优化**:
  - 为 user_id + platform 创建联合索引
  - 分页查询限制每页最多 100 条
- **缓存策略**:
  - 账号配置缓存到 Redis,TTL 30 分钟
  - 更新/删除时主动清除缓存
```

**关键点:**
- ✅ 自主补充了完整的数据库设计

## 7. 危险信号识别

### 危险信号 1:盲目假设技术细节

| 想法 | 现实 | 正确做法 |
|------|------|----------|
| "让我搜索代码库" | 代码可能错误。向用户请求参考资料。 | 停止。使用 AskUserQuestion 请求参考信息。 |
| "我假设我们使用 X 技术" | 假设会失败。先向用户确认。 | 停止。询问项目使用的技术栈。 |
| "这可能是这样工作的" | 可能 ≠ 确定。请求明确信息。 | 停止。请求用户提供机制说明。 |

### 危险信号 2:过度询问标准设计元素

| 想法 | 现实 | 正确做法 |
|------|------|----------|
| "我是否需要添加错误码定义?" | 这是标准设计元素。应该自主补充。 | 直接在文档中补充完整的错误码定义。 |
| "我是否应该补充日志记录策略?" | 这是标准设计元素。应该自主补充。 | 直接在文档中补充日志设计。 |
| "我需要了解 5 个技术点" | 一次问太多。先问 1-2 个最关键的。 | 拆分问题,每次只问 1-2 个。 |

### 危险信号 3:偏离或遗忘 PRD

| 想法 | 现实 | 正确做法 |
|------|------|----------|
| "我记得 PRD 说的是..." | 记忆可能错误。必须查看原文。 | 停止。立即重新阅读 PRD 相关章节。 |
| "PRD 应该是这个意思" | 不要猜测。以 PRD 原文为准。 | 停止。查看 PRD 原文,或询问用户。 |
| "设计完成了,可以输出了" | 必须先对照 PRD Review。 | 停止。执行 PRD 对照检查。 |

### 危险信号 4:缺少参考信息

| 想法 | 现实 | 正确做法 |
|------|------|----------|
| "设计完成了,可以输出了" | 必须先添加参考资料。 | 停止。添加所有用到的参考信息。 |
| "参考资料不重要" | 参考资料确保可追溯性。 | 停止。记录所有链接、文档路径、代码路径。 |

## 总结

**关键区分:**
- **技术不确定性** → 停止并询问用户
- **标准设计元素** → 获得充分上下文后自主补充
- **PRD 内容** → 必须严格遵循,有疑问立即查看原文
- **参考资料** → 必须完整记录,确保可追溯
- **证据扫描** → 适度扫描,及时澄清,避免盲目探索
- **图表使用** → 按需使用,图文结合,避免过度复杂
