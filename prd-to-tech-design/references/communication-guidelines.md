# 沟通指南：如何有效询问用户

本文档提供与用户沟通的详细指南，帮助高效收集技术不确定性信息。

## 核心原则

### 原则 1：拆分问题，每次只问 1-2 个

**为什么？**
- 用户面对大量问题会感到压力
- 回答质量下降或直接放弃
- 难以聚焦关键问题
- 效率低下

**如何做？**
1. 识别所有需要了解的技术点
2. 按优先级排序（最阻塞设计的排在前面）
3. 每次只问 1-2 个最关键的问题
4. 等用户回答后再问下一批

**示例：**

❌ **错误做法**：
```
我需要了解以下 8 个技术点：
1. 账号登录器机制
2. 内容发布机制
3. 代理配置机制
4. 定时任务调度机制
5. BOS 存储机制
6. 消息队列机制
7. 缓存策略
8. 监控告警机制

请逐一回答这些问题。
```

✅ **正确做法**：
```
PRD 提到"唤起账号登录器"，这是设计的核心部分。我需要了解：
1. 登录器是什么？（前端组件/独立客户端/浏览器插件？）
2. 如何唤起登录器？（API 调用/前端事件/其他方式？）

请提供相关说明或文档路径。

[等用户回答后，再问下一批问题]
```

### 原则 2：每个问题要清晰、具体、易于回答

**为什么？**
- 宽泛的问题难以回答
- 用户不知道从何说起
- 回答可能不是你需要的信息

**如何做？**
1. 聚焦一个具体的技术点
2. 提供选项或示例帮助用户理解
3. 避免复合问题
4. 使用用户熟悉的术语

**示例：**

❌ **错误做法**：
```
请介绍一下账号管理系统的整体架构。
```
（太宽泛，用户不知道从何说起）

✅ **正确做法**：
```
PRD 提到"唤起账号登录器"，请问登录器是前端组件还是独立客户端？
```
（具体明确，易于回答）

### 原则 3：明确说明为什么需要这些信息

**为什么？**
- 让用户理解问题的重要性
- 提高用户回答的积极性
- 帮助用户提供更准确的信息

**如何做？**
1. 说明这个信息对设计的影响
2. 解释为什么需要了解这个技术点
3. 让用户理解问题的上下文

**示例：**

❌ **错误做法**：
```
请问代理配置在哪个层级？
```

✅ **正确做法**：
```
为了设计代理配置的存储方案，我需要了解：
代理配置是在账号级别（每个账号独立配置）还是平台级别（一个平台共享配置）？

这会影响数据库表结构的设计。
```

### 原则 4：提供多种信息提供方式

**为什么？**
- 用户可能更擅长某种方式
- 降低用户的回答门槛
- 提高信息收集效率

**如何做？**
1. 文字说明
2. 文档路径
3. 代码路径
4. 图片
5. URL

**示例：**

✅ **正确做法**：
```
PRD 提到"唤起账号登录器"，我需要了解登录器的工作机制。

请提供以下任一方式的信息：
- 文字说明：登录器是什么、如何工作
- 文档路径：相关的技术文档
- 代码路径：关键的代码文件
- 图片：流程图或架构图
```

### 原则 5：用户明确提到的文档直接读取

**为什么？**
- 避免重复确认
- 提高效率
- 用户已经明确指出了参考资料

**如何做？**
- 如果用户说"参考 docs/xxx.md"，直接使用 Read 工具读取
- 无需再次确认"我可以读取这个文档吗？"

**示例：**

用户：
```
登录器的实现参考 docs/login-widget.md
```

❌ **错误做法**：
```
我可以读取 docs/login-widget.md 这个文档吗？
```

✅ **正确做法**：
```
[直接使用 Read 工具读取 docs/login-widget.md]
好的，我已经阅读了登录器的实现文档。
```

## 何时询问 vs 何时自主补充

### 必须询问用户的情况（技术不确定性）

这些信息只有用户知道，必须询问：

1. **现有系统的具体实现机制**
   - 示例："账号登录器如何工作？"
   - 示例："内容发布到小红书的流程是什么？"
   - 示例："代理配置在哪个层级生效？"

2. **技术选型的可行性**
   - 示例："项目是否使用 APScheduler？"
   - 示例："APScheduler 是否支持动态任务调度？"
   - 示例："是否需要考虑其他任务调度方案？"

3. **业务规则的具体定义**
   - 示例："代理配置是账号级别还是平台级别？"
   - 示例："定时发布的触发时机是什么？"
   - 示例："账号删除后关联数据如何处理？"

4. **架构决策**
   - 示例："单体应用还是微服务？"
   - 示例："是否需要拆分服务？"
   - 示例："前后端如何通信？"

5. **性能指标的具体要求**
   - 示例："QPS 要求是多少？"
   - 示例："接口响应时间要求是多少？"
   - 示例："并发用户数要求是多少？"

6. **项目特定的安全规范和约束**
   - 示例："是否有特定的安全规范文档？"
   - 示例："敏感数据如何加密？"
   - 示例："是否需要符合特定的合规要求？"

### 应该自主补充的内容（标准设计元素）

这些是通用的标准设计元素，获得充分上下文后应该自主补充：

1. **标准的错误处理机制**
   - 统一的错误码定义
   - 错误响应格式
   - 异常捕获和降级处理

2. **完整的日志设计**
   - 关键操作的日志记录点
   - 日志级别定义
   - 敏感信息脱敏规则

3. **监控和告警指标**
   - 接口响应时间监控
   - 错误率监控
   - 关键业务指标

4. **数据校验规则**
   - 必填字段校验
   - 数据格式校验
   - 数据长度限制

5. **接口的标准设计元素**
   - 分页参数
   - 排序参数
   - 过滤参数

6. **数据库设计的标准字段**
   - 主键、创建时间、更新时间
   - 软删除标记
   - 创建人/更新人

7. **安全防护措施**
   - 输入验证和 SQL 注入防护
   - XSS 防护
   - 接口鉴权和权限控制

8. **性能优化方案**
   - 数据库查询优化
   - 缓存策略
   - 异步处理

详细的标准设计元素清单请参考 `references/standard-design-elements.md`。

## 问题优先级排序

### 高优先级问题（必须先问）

这些问题阻塞核心设计，必须先问：

1. **核心业务流程的实现机制**
   - 如果不了解，无法设计核心功能
   - 示例：登录器如何工作、内容如何发布

2. **关键技术选型的可行性**
   - 如果选型错误，需要重新设计
   - 示例：任务调度框架是否支持动态调度

3. **数据模型的关键决策**
   - 如果理解错误，数据库设计需要重构
   - 示例：代理配置在哪个层级

### 中优先级问题（可以稍后问）

这些问题影响设计细节，可以在基本设计完成后再问：

1. **性能指标的具体要求**
   - 可以先按常规标准设计，后续优化
   - 示例：QPS 要求、响应时间要求

2. **非核心功能的实现细节**
   - 不影响核心设计
   - 示例：监控告警的具体方案

### 低优先级问题（可选）

这些问题可以在设计基本完成后，作为"待确认问题"列出：

1. **灰度发布方案**
   - 可以在部署阶段再确定
   - 示例：灰度策略、灰度比例

2. **降级和容灾方案**
   - 可以在上线前再完善
   - 示例：降级开关、容灾演练

## 提问模板

### 模板 1：了解现有系统机制

```
PRD 提到"{功能描述}"，为了设计{相关模块}，我需要了解：
1. {具体问题 1}？
2. {具体问题 2}？

请提供：文字说明、文档路径、或关键代码文件路径
```

### 模板 2：确认技术选型可行性

```
PRD 需要{功能描述}，我需要确认技术选型：
1. 项目中是否已使用{技术框架}？
2. {技术框架}是否支持{具体需求}？

请提供：现有实现方式或相关文档
```

### 模板 3：确认业务规则

```
为了设计{模块名称}，我需要确认业务规则：
{具体问题}？

这会影响{设计影响}。
```

### 模板 4：确认架构决策

```
PRD 涉及{功能范围}，我需要确认架构方案：
1. {架构问题 1}？
2. {架构问题 2}？

请提供您的建议或项目的架构规范。
```

## 沟通流程示例

### 第一轮沟通：核心机制

```
AI：我已阅读 PRD，识别出以下核心技术点需要了解：

PRD 提到"唤起账号登录器"，这是设计的核心部分。我需要了解：
1. 登录器是什么？（前端组件/独立客户端/浏览器插件？）
2. 如何唤起登录器？（API 调用/前端事件/其他方式？）

请提供：文字说明、文档路径、或关键代码文件路径
```

### 第二轮沟通：技术选型

```
用户：登录器是一个独立的前端组件，通过调用 window.openLoginWidget() 触发。
参考文档：docs/login-widget.md

AI：感谢提供信息。我还需要确认技术选型：

PRD 提到定时发布功能，我需要确认：
1. 项目中是否已使用任务调度框架（如 APScheduler、Celery）？
2. 如果使用 APScheduler，是否支持动态读取数据库并调度任务？

请提供：现有任务调度的实现方式或相关文档
```

### 第三轮沟通：业务规则

```
用户：项目使用 APScheduler，参考代码：src/scheduler/task_manager.py

AI：好的，我已经获得了足够的技术上下文。现在开始编写技术设计文档。

[开始编写设计文档，自主补充标准设计元素]
```

## 常见沟通错误

### 错误 1：一次性问太多问题

❌ 列出 5-10 个问题让用户逐一回答

✅ 每次只问 1-2 个最关键的问题

### 错误 2：问题太宽泛

❌ "请介绍一下系统架构"

✅ "登录器是前端组件还是独立客户端？"

### 错误 3：反复询问标准设计元素

❌ "我是否需要添加错误码定义？"

✅ 直接在设计文档中补充错误码定义

### 错误 4：没有说明为什么需要这些信息

❌ "请问代理配置在哪个层级？"

✅ "为了设计代理配置的存储方案，我需要了解代理配置是在账号级别还是平台级别？"

### 错误 5：重复确认用户已提供的信息

❌ 用户说"参考 docs/xxx.md"，还问"我可以读取这个文档吗？"

✅ 直接读取文档，无需再次确认
